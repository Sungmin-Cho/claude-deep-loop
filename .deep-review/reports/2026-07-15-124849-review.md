### 🟡 Yellow — Concerns requiring response

1. **Valid `recordCost` callers still omit request IDs / solo 1-of-2**
   - **Source**: Codex standard P1; independently reproduced across `tests/budget.test.mjs` and
     `tests/insights.test.mjs`.
   - **Evidence**: Task 7C makes `requestId` mandatory but its caller migration leaves several
     successful, fence, reconciliation, floor, and insights calls unchanged. They fail before their
     intended assertions, so the full suite cannot pass.
   - **Required response**: provide a literal complete migration for every valid call, preserve only
     the two invalid-cost precedence calls without authority, and prove closure mechanically.

2. **The accounting caller-closure assertion rejects a valid variable fence / solo 1-of-2**
   - **Source**: Codex standard P1; independently reproduced from the Task 7C headless-host diff and
     closure regex.
   - **Evidence**: the final maker-process call passes `fence: accountingFence`, while the assertion
     accepts only an inline object beginning with `fence: { ... owner:`.
   - **Required response**: either inline that fence or statically prove the variable's current-fence
     provenance, then run the closure test against every actual call.

3. **Nested review dispatch omits complete episode input from its intent / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently traced through Task 7F and the Task 11B closure
     table.
   - **Evidence**: `dispatchReview` hashes point, workstream, subagent, and detected capabilities, then
     reuses that mutation context for episode creation. Target maker, reviewer resolution, evidence,
     contract, and blocked reason are absent, so changed episode input can authenticate and recover an
     old marker before the response projection rejects it.
   - **Required response**: derive the complete normalized episode request before gateway entry and
     bind it into the outer review intent, or give episode creation its own non-nested complete intent.

4. **Episode creation lacks post-cleanup idempotency / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced from `episodeAppend`.
   - **Evidence**: only `context.recovered` invokes `onRecovered`; after marker cleanup an exact retry
     always appends a new episode, and the projection chooses `episodes.at(-1)` rather than a durable
     request-correlated episode.
   - **Required response**: persist a bounded episode request digest, locate exactly one matching
     durable episode/event on ordinary retry, and prove dispatch cannot launch a duplicate checker.

5. **Three receipt settlement replays omit their current operation policy / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced from the four replay-fence wrappers.
   - **Evidence**: only generic `recordCost` sets `leasePolicy:true`. Preflight, process, and terminal
     maker settlement verify owner/generation/runtime but can return durable success before their
     terminal, lease-state, or narrow accounting carve-out policy executes.
   - **Required response**: move each writer's exact current policy into its replay fence and test
     terminal/released/paused and allowed carve-out precedence without changing intended idempotency.

6. **Generic accounting durable identity omits runtime and intent / solo 1-of-2**
   - **Source**: Codex adversarial Yellow; independently checked against design §6.1.
   - **Evidence**: the event stores only a digest of `requestId`; durable replay compares amounts,
     owner, and generation. Reusing the ID with equal amounts but changed runtime or intent can return
     the old success whenever the new fence also passes.
   - **Required response**: persist a domain-separated digest over request ID plus owner, generation,
     amounts, runtime, and intent, retain a separate ID digest to detect altered reuse, and test both
     conflict dimensions.
