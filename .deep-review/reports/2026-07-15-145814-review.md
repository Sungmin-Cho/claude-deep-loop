## Findings

### 🔴 Red — Independently corroborated blockers

1. **Review dispatch has no stable logical request identity / 2-of-2**
   - **Sources**: Codex standard P1; Codex adversarial Yellow.
   - **Evidence**: `dispatchRequestDigest` is derived only from owner/generation plus point,
     workstream, detection map, and subagent flag. A rejected checker followed by a done fix maker in
     the same generation produces the same digest, so the second intentional round collides with the
     first. Conversely, a response-loss retry after a newer maker appears can move the same digest to
     that newer maker.
   - **Required response**: require a bounded caller-generated dispatch request ID, store separate
     ID/full-request digests, perform global ID replay before maker selection, reject changed reuse,
     and rotate the ID for every intentional review round.

2. **Episode correlation trusts a copied digest instead of the immutable projection / 2-of-2**
   - **Sources**: Codex standard P1; Codex adversarial Red.
   - **Evidence**: `verifyEpisodeCreationCorrelation` compares the state object's stored
     `creation_request_digest` with the event copy and checks only plugin/role/kind/point plus initial
     status/reason. It never recomputes the digest from workstream, expected artifacts, target maker,
     reviewer resolution, evidence, contract, and creation identity. A hash-valid state-only rewrite
     can therefore redirect checker proof to a different maker while retaining the copied digest.
   - **Required response**: reconstruct the complete canonical creation projection from each durable
     episode object, recompute its domain-separated digest, compare it to the exact creation event,
     and add semantic-field corruption tests.

### 🟡 Yellow — Concerns requiring response

1. **Human breaker reset never supplies its newly required request ID / solo 1-of-2**
   - **Source**: Codex standard P1; independently reproduced from the current CLI handler and Task 7G
     afterimage.
   - **Evidence**: planned `resetBreaker` rejects missing `requestId`, while `breaker reset --confirm`
     still calls it with `{ fence }`; status/continue instructions also expose no request-ID argument.
   - **Required response**: make `--request-id` mandatory on the human CLI boundary, document
     allocate-once/reuse-on-response-loss semantics, migrate every command example/test, and preserve
     exit-2 usage versus exit-3 fence precedence.

2. **Final review-dispatch lock does not recheck checker eligibility / solo 1-of-2**
   - **Source**: Codex standard P2; independently reproduced from the two-phase dispatch patch.
   - **Evidence**: two different requests can both pass the first read with no checker and then create
     distinct active checkers because the second lock only calls episode creation and never repeats
     the latest-maker/active-or-terminal checker CAS.
   - **Required response**: globally replay the same ID first, then rederive the selected latest maker
     and enforce one proof-capable checker in the final lock before append.

3. **Missing request rematerialization can still replace a concurrent execution-plane edit / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced from `existsSync` followed by
     replacement-style `atomicWrite`.
   - **Evidence**: the run lock does not serialize external editing. If the request is absent at the
     check but created before rename, unconditional rename replaces the new editable bytes.
   - **Required response**: use a durable atomic create-if-absent/no-replace primitive, preserve the
     winner on `EEXIST`, clean only the caller's temp, and exercise the check-to-publish race.

4. **Production same-file identity can compare rounded inode numbers / solo 1-of-2**
   - **Source**: Codex adversarial Yellow; independently reproduced from all three production
     `statSync(value)` dependency factories.
   - **Evidence**: ordinary numeric `dev`/`ino` values are compared directly even though filesystem
     identifiers can exceed JavaScript's safe-integer range, allowing distinct identities to round to
     the same number.
   - **Required response**: use `statSync(path, { bigint: true })` for production identity and add
     large adjacent BigInt fixtures across POSIX/Windows routing paths.

5. **The disposable afterimage validator syntax-checks but never runs installed behavior / solo 1-of-2**
   - **Source**: Codex adversarial Yellow; independently reproduced from the validator tail.
   - **Evidence**: Task 7G production afterimages are installed and `node --check`ed; selected caller
     diffs are applied; the tree is then deleted without `node --test` or an equivalent executable
     probe against those bytes.
   - **Required response**: run focused tests/probes inside the disposable tree after all production,
     helper, caller, and test migrations are installed; fail the plan validator on nonzero exit.

6. **`recordReviewVerdict` drops the existing full lease policy / solo 1-of-2**
   - **Source**: Codex adversarial Yellow; independently reproduced from current `breaker.mjs`,
     `leaseCheck`, and the Task 7G afterimage.
   - **Evidence**: the shared gateway authenticates only owner/generation. Unlike trip/reset, verdict
     recording never performs the current runtime/released/releasing/paused/terminal lease check
     before replay or fresh append, so a released same-generation owner can still mutate breaker
     state.
   - **Required response**: retain identity-first authentication, then execute the complete current
     business lease policy before durable replay/fresh semantics and test released/releasing/paused
     rejection with zero writes.
