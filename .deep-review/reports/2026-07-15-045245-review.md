## Findings

### 🔴 Red — Critical / blocking

1. **deadline authority and non-reentrant locking / corroborated — the final plan has no coherent lock-fresh clock contract**
   - **Sources**: Codex standard P1 and Codex adversarial Yellow; main-agent line-level reproduction.
   - **Location**: design `554-560` and `656-661`; plan `10901-10923`, `23334-23368`, and
     `23462-23528`.
   - **Evidence**: the design first requires the injectable `nowFn` to be sampled after lock
     acquisition, but later says caller-provided clock callbacks run outside every mutation context.
     Task 11C samples `decisionNow` before `appCommitPhase`, after which the final locked append is
     explicitly given `nowFn: () => decisionNow`. Descriptor/gate/reconciliation work or lock wait can
     cross `prepare_deadline`, yet the commit reuses the earlier timestamp and authorizes a stale App
     action. Conversely, confirm/revoke pass the callable clock into the locked gateway and sweep calls
     it inside its commit context, so a reentrant injected clock violates the promised callback safety.
   - **Failure scenario**: a prepare begins before its deadline, performs slow external work, and
     acquires the final lock after expiry. The fixed stale sample still passes the deadline check and
     publishes an actionable App descriptor. A test clock that performs the advertised verified-read
     reentry instead collides with the non-reentrant lock.
   - **Required response**: define one explicit clock boundary. The final mutation must sample a
     gateway-owned, documented non-reentrant clock source only after acquiring the lock; reentrant or
     external callbacks remain outside. Do not pass a pre-lock fixed sample back as the authoritative
     clock. Apply the rule to prepare/confirm/revoke/sweep and add slow-precommit plus reentrant-clock
     tests that prove expiry cannot be authorized and lock callbacks cannot reenter.

### 🟡 Yellow — Concerns requiring response

1. **Windows persistence / solo 1-of-2 — durable file fsync reopens the file read-only**
   - **Source**: Codex standard P1; independently verified against the literal plan and platform
     contracts.
   - **Location**: plan `4957-4963`; Task 15A's real `windows-latest` durability cell.
   - **Evidence**: `syncRegularFile` opens every stage with Node flag `'r'` and then calls
     `fsyncSync`. Node maps `'r'` to read-only access on Windows, while Win32
     `FlushFileBuffers` requires a handle with `GENERIC_WRITE` access. The universal genesis/journal
     publisher therefore cannot complete its mandatory file flush on native Windows.
   - **Required response**: retain and flush the original writer descriptor or reopen without
     truncation using write-capable access such as `'r+'`; add a real Windows assertion that the file
     flush succeeds before rename, not only a dependency-mocked call-order test.

2. **sequential task composition / solo 1-of-2 — Task 12B dereferences Task 11C's snapshot wrapper as a loop**
   - **Source**: Codex standard P1; independently verified against the final task order.
   - **Location**: plan `23441-23445` and `24197-24217`.
   - **Evidence**: Task 11C changes `snapshot` to `{data,hash}` and introduces
     `const loop = snapshot.data`. Task 12B, which runs later, still reads `snapshot.project` and
     `snapshot.session_chain` and passes `loop: snapshot` to the descriptor builder. Applied in order,
     production prepare throws before returning its first public App action.
   - **Required response**: rewrite Task 12B against the final `loop` local and current child/
     continuation bindings, and make the plan validator compose Tasks 11C→12B into one syntax/runtime
     probe rather than validating each fence in isolation.

3. **recovery fencing / solo 1-of-2 — receipt mutation intents omit the authoritative stdin mode**
   - **Source**: Codex standard P2; independently verified against the gateway's recovery-before-body
     ordering.
   - **Location**: design `637-645`; plan `11027-11030`, `11051-11058`, and the confirm/fail final
     implementations.
   - **Evidence**: `app-confirm` binds only the receipt digest; `app-fail` binds failure/reason/receipt,
     but neither binds `stdinMode`. After marker publication, a same-receipt retry with a different
     mode has the same intent and is allowed to recover canonical state before
     `assertRecordedReaderMode` can return the public fence.
   - **Required response**: include normalized `stdin_mode` in both receipt-bearing intents (null for
     non-receipt failure), and test marker-published wrong-mode confirm and `message-unconfirmed`
     retries for exit 3 plus byte invariance before exact recovery.

4. **runtime fencing / solo 1-of-2 — acquire recovery intent omits runtime**
   - **Source**: Codex standard P2; independently verified against the same recovery ordering.
   - **Location**: design `637-645`; plan `11035-11038` and the acquire entry fence.
   - **Evidence**: the acquire intent binds child, observation, and stdin mode but not `runtime`. A
     marker created by a Codex acquire can therefore be recovered by an otherwise identical
     `runtime=claude` retry before the body rejects it as `RUNTIME_FENCED`, mutating lease ownership for
     a rejected caller.
   - **Required response**: bind normalized runtime into `app-acquire`, and add a crash-after-marker
     wrong-runtime retry proving the marker/canonical bytes do not change until the exact Codex retry.

5. **crash-test executability / solo 1-of-2 — the composed worker cannot reach the recovery seams**
   - **Source**: Codex adversarial Red; independently reproduced with exact source predicates.
   - **Location**: plan `10259-10272`, `19522-19550`, `19626-19634`, `19735-19744`,
     `19765-19774`, `19814-19823`, and `20453-20462`.
   - **Evidence**: the worker rejects any process lacking `DEEP_LOOP_CRASH_OWNER` and
     `DEEP_LOOP_CRASH_GENERATION`; every Task 10D/11A spawn shown above sets only
     `DEEP_LOOP_CRASH_INPUT`, so it exits 1 before dispatch instead of the asserted 91. Even after the
     environment fix, worker prepare/confirm omit the dated fixture's fixed clocks, and prepare also
     omits its deterministic gate/reconciliation seams, so current wall time rejects the July 13
     attempt before journal publication.
   - **Required response**: pass both required environment values on every spawn; mirror the fixture's
     fixed clock/gate/reconciliation dependencies in the extension dispatch; add a preliminary per-
     operation assertion that exit 91 and the expected pending marker were actually reached before any
     recovery/fencing claim.

6. **fixed interface contract / solo 1-of-2 — `appendAnchored` omits mandatory options consumed by its implementation**
   - **Source**: Codex adversarial Yellow; independently verified.
   - **Location**: plan `156-158`, `240-242`, and `10920-10962`.
   - **Evidence**: the authoritative fixed signature omits `intentDigest` and `allowTerminal`, while the
     literal implementation rejects an absent digest with `MUTATION_INTENT_REQUIRED` and consumes
     `opts.allowTerminal` to authorize proof-bearing terminal paths. Later cards are told to use the
     incomplete fixed shape.
   - **Required response**: add both options to the fixed declaration and make the embedded validator
     compare declared option names with every mandatory/consumed literal implementation option.
