## Findings

### 🔴 Red — Blocker

1. **The legacy proof transition is unreachable before checkpoint and unusable after it / 2-of-2**
   - **Sources**: Codex standard P1; Codex adversarial Red; independently reproduced.
   - **Evidence**: with no baseline, `verifyProofTransitionCorrelation` scans from event index zero
     and requires new `proof_transitions` on ordinary 1.8.2 `episode-*` and `workstream-*` events.
     Verified reads therefore fail before `legacyCheckpointSpec` can append the first checkpoint.
     If a checkpoint is present, verification recomputes its full legacy proof digest from current
     state and requires the original entity prefix to remain byte-equivalent forever. The same
     upgrading transaction consequently cannot record/abandon an existing episode, change an
     existing workstream status or active membership, or add `review_points_done`.
   - **Required response**: treat pre-checkpoint history as opaque, but seed authenticated transition
     origins for every existing proof entity in the baseline. Subsequent mutations of legacy and
     new entities must advance the same canonical per-entity chain rather than freezing a prefix.
     Test an in-progress legacy maker through record, review, workstream completion, and finish, as
     well as missing/substituted origin and transition attacks.

### 🟡 Yellow — Concerns requiring response

1. **Noncanonical project-root aliases can commit an episode without its request file / solo 1-of-2**
   - **Source**: Codex standard P2; independently reproduced from the literal path construction.
   - **Evidence**: episode state persists `request_path` using `runDir(root, runId)`, while later
     materialization resolves the run directory and requires equality with that canonical sibling.
     A valid symlink alias therefore commits state/event and then throws
     `EPISODE_MATERIALIZATION_WORK_INVALID` before publishing the request file.
   - **Required response**: derive and persist the request path from the same canonical run identity
     before the transaction; add a symlink-root retry/materialization test proving one state entity
     and one durable request file.

2. **The proof-transition writer retrofit is prose-only and absent from the installed afterimage / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced from Task 7F and the validator.
   - **Evidence**: Task 7F promises transition wiring for review, workspace, and state writers but
     supplies only prose for nine writer migrations and a table-driven test. Its RED command does
     not select a proof-transition test, and sequential installation excludes `workspace.mjs` and
     `state.mjs`. Token-presence assertions can therefore report `ok:true` while the promised
     production afterimage is not installed or exercised.
   - **Required response**: centralize transition derivation in the prospective mutation gateway or
     provide complete writer diffs. The sequential validator must install every affected production
     file and execute exact creation/mutation/recovery/tamper tests against the installed tree.

3. **`active_workstreams` bytes are not exactly represented by the proof projection / solo 1-of-2**
   - **Source**: Codex adversarial Yellow; independently reproduced.
   - **Evidence**: each workstream projection reduces active state to a Boolean. Duplicate or unknown
     IDs can change the stored array and operational `max_parallel` behavior without changing any
     Boolean projection, while `state-patch` may carry an empty transition set.
   - **Required response**: schema-enforce a canonical unique array of existing workstream IDs and
     bind the exact ordered/canonical active-membership projection. Add duplicate, unknown, reorder,
     removal, and addition probes.

4. **Caller-supplied journal `crashProbe` callbacks remain reachable under the run lock / solo 1-of-2**
   - **Source**: Codex adversarial Yellow; independently reproduced.
   - **Evidence**: exported mutation APIs accept `crashProbe`, the journal publisher invokes it
     between durable steps, and the verified mutation context still owns the non-reentrant lock.
     Prose that production callers omit the option does not prevent an internal reentrant caller
     from reaching `LOCK_BUSY` or performing arbitrary lock-held work.
   - **Required response**: replace function-valued public seams with a closed non-callable fault
     schedule interpreted by the gateway, or confine function probes to an unexported test harness
     that cannot re-enter production APIs. Test that public inputs cannot execute lock-held code.

5. **Two older evidence-ledger receipts remain absent from git / solo 1-of-2**
