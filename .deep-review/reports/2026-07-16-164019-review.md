## Findings

### 🟡 Yellow — Required concerns

1. **The displayed worker URLs are passed to `spawnSync` through Windows-incompatible `.pathname`**
   - **Source**: Codex standard P1; independently reproduced.
   - **Evidence**: Task 5B creates a `URL` and passes `INIT_TRANSACTION_CRASH_WORKER.pathname`
     directly to Node (`plan:5029-5034`). Task 6A repeats `worker.pathname` three times
     (`plan:6572`, `6627`, `6661`). For `file:///C:/Program%20Files/...`, Node reports pathname
     `/C:/Program%20Files/...`, while `fileURLToPath(url, { windows:true })` returns the actual
     `C:\Program Files\...` path. The required Windows CI cells therefore cannot reliably find a
     worker under a spaced or encoded checkout path.
   - **Required response**: import `fileURLToPath`, derive one filesystem path for every worker URL,
     use it at every spawn, add a Windows URL/path regression, and make the validator reject
     `.pathname` process arguments.

2. **A genesis temp `wx` collision deletes a pathname the invocation never owned**
   - **Source**: Codex standard P2; independently executed.
   - **Evidence**: `writeGenesisArtifact` performs a `writeFile(...,{flag:'wx'})` inside an
     unconditional `try/finally`, and always calls `removeOwnTemp(temp,deps)` (`plan:5634-5644`).
     If `wx` throws `EEXIST`, an exact extraction of the displayed function invoked the injected
     `unlink` on that pre-existing pathname before rethrowing the original collision. This violates
     the card's foreign-artifact preservation contract and can erase another/crashed initializer's
     evidence.
   - **Required response**: track temp ownership like the lock candidate path, skip cleanup on
     `EEXIST`, clean only a successfully created or possibly partial own unique temp, and execute a
     pre-existing-byte preservation regression in both the task tests and embedded validator.

3. **Task 5B crash recovery expects `LOCK_STALE_MANUAL` while injecting an `alive` holder**
   - **Source**: Codex adversarial Red; independently executed.
   - **Evidence**: the scalar worker hard-exits 91 and strands a held authority (`plan:5032-5038`).
     Both recovery tests then build deps with `probePidIdentity: () => 'alive'`
     (`plan:5097-5106`, `5151-5165`) and expect `LOCK_STALE_MANUAL`. Exact `withInitLock` execution
     on a held authority with that injected result throws `LOCK_BUSY`, as required by
     `throwHeldAuthority` (`plan:4860-4868`). The tests fail before compaction/recovery, and the
     embedded scalar probe covers only durable-file exit 91 rather than this liveness contract.
   - **Required response**: use the exact crashed child PID/identity and a definitively-dead result
     for stale-manual recovery tests, preserve separate live-holder BUSY coverage, and make the
     embedded validator execute the displayed worker→stale-authority→approved-compaction→retry
     path rather than only a scalar durable-file exit.

4. **The write-free 65th cap guarantee is lost when another contender takes the last slot between traversals**
   - **Source**: Codex adversarial Yellow; independently executed against the exact lock fence.
   - **Evidence**: contender A can pass the initial traversal at 63 released authorities
     (`plan:4879-4882`). During A's candidate write, contender B can publish and release authority
     64. A then writes its candidate, the second traversal raises `LOCK_CHAIN_EXHAUSTED`, and A's
     `finally` unlinks its candidate (`plan:4895-4905`, `4949-4951`). The independent probe observed
     exactly one A write and one A unlink. The sequential cap tests (`plan:4675-4699`,
     `36561-36583`) cannot expose this last-slot overtaking.
   - **Required response**: replace the racy two-traversal precheck with a reservation/publication
     protocol that cannot perform sweep/candidate mutation after the cap becomes visible, or narrow
     and justify the contract without claiming concurrent write-free failure. Add the exact
     63-owner/A-precheck/B-owner64/A-resume interleaving to the card and validator.
