## Findings

### 🔴 Red — Critical / blocking

1. **recovery / corroborated — valid journal states can wedge or publish beside unknown debris**
   - **Sources**: Codex standard P1, Codex adversarial Red, and main-agent independent verification.
   - **Location**: plan `10059-10068`, `10204-10212`, `10245-10258`, `10285-10343`;
     current legacy genesis `scripts/lib/initrun.mjs:53-56`.
   - **Evidence**:
     1. a legacy first transaction can have no canonical event log; old-state/old-hash recovery calls
        `truncateSync` on that absent path and repeats `ENOENT` forever;
     2. `durableReplace` uses a fixed `<canonical>.replace` with `flag:'wx'`, so a crash after its
        creation but before rename leaves debris that every exact retry rejects with `EEXIST`;
     3. with a valid marker present, recovery reads only the four fixed paths and never scans for an
        extra or symlinked `.anchored-*` artifact. It can publish and clean the known journal while
        leaving the forbidden artifact behind, contrary to the promised no-repair corruption test.
   - **Required response**: support absent-before-log recovery, make inner replace steps resumable,
     reject every unknown/symlink journal artifact before canonical writes, and add crash/negative
     fixtures inside canonical state/hash replacement as well as the outer eight probes.

2. **idempotency — generic mutation intent omits request-distinguishing inputs**
   - **Source**: Codex adversarial Red, independently verified.
   - **Location**: plan `10632-10639`; current `scripts/lib/state.mjs:120-127` and
     `scripts/lib/workspace.mjs:13-21,71-78`.
   - **Evidence**: the public generic fallback hashes only event type/data. `state-patch` deliberately
     omits `value`, while `workstream-new` records only `title` and omits branch/worktree/base/deps.
     A same-fence retry with the same event data but different business input recovers and publishes
     the first request, then can execute the second request as another mutation.
   - **Required response**: every public mutation must provide an operation-specific, domain-separated,
     raw-free digest of all request-distinguishing inputs, with generic same-caller/different-input
     crash tests.

3. **execution / verified — App emit uses `emitFence` outside its lexical scope**
   - **Source**: Codex adversarial Red.
   - **Location**: plan `14803-14820`, then references at `14919`, `14954`, and `15108`.
   - **Evidence**: `emitFence` is declared inside the reservation callback but used after that callback
     returns. A fresh emit can reserve and then throw `ReferenceError`, leaving an active reservation;
     exact retry follows the same broken path.
   - **Required response**: define the fence in the enclosing `emitHandoff` scope and add literal
     execution coverage through reserve, descriptor construction, and final publication.

4. **locking / verified — prepare, confirm, revoke, and sweep race callbacks run under the run lock**
   - **Source**: Codex standard P1.
   - **Location**: plan `15942-16049`, `16590-16635`, `17219-17275`, `22609-22631`.
   - **Evidence**: the four functions still wrap their complete bodies in `withAppMutation`; inserting
     `beforeAppendFn` immediately before `mutation.appendAnchored` therefore keeps the non-reentrant
     lock held. Two-worker barriers wait for peers that cannot acquire the lock, producing a bounded
     deadlock instead of the asserted CAS race.
   - **Required response**: split the verified snapshot and final commit into short contexts and run
     callbacks between them, with a fresh final-lock CAS proof.

5. **artifact recovery / verified — insights retry can report success before publishing its artifact**
   - **Source**: Codex standard P1.
   - **Location**: plan `13100-13150`.
   - **Evidence**: a crash after the anchored `insights-emitted` commit but before the final rename
     leaves only `.tmp-<ulid>`. The recovered entry sees the event and returns success immediately,
     never validating or renaming the staged artifact; `latestInsights` cannot consume the referenced
     final path.
   - **Required response**: bind staged/final artifact identity into the operation and finish or
     fail-closed the artifact publication before returning recovered success.

6. **privacy / verified — `finish` control events do not have an exact key set**
   - **Source**: Codex adversarial Red.
   - **Location**: plan `8767-8803`, `8979-8991`, `9232-9243`, `19447-19451`.
   - **Evidence**: `finish` is an App control type but absent from `APP_CONTROL_EXACT_KEYS`; global
     terminal validation checks values, not extra keys. A hash-valid rewritten event can add
     `thread_id` or another raw receipt and still pass the proposed verifier.
   - **Required response**: add exact generic/App-bound finish key sets and hash-valid extra-field
     negatives for both shapes.
