## Findings

### 🔴 Red — Blockers

1. **Episode request publication has no stable containment/transaction boundary / 2-of-2**
   - **Sources**: Codex standard P1; Codex adversarial Yellow 1 and Yellow 2; independently reproduced.
   - **Evidence**: `canonicalEpisodeRequestPath` calls `realpathSync(runDir(...))` from lock-held
     recovery and precheck, contrary to the design's final-lock scalar-only rule. The actual file is
     then published after the state/event lock is released. A concurrent run-directory replacement
     between verified read and `createFileDurablyIfAbsent` can redirect the temporary write outside
     `.deep-loop` before parent validation. The alias test makes only one call and therefore does not
     exercise response loss after state/event commit or same-ID rematerialization.
   - **Required response**: give request bytes one authenticated recoverable publication boundary
     that holds a stable run-directory identity and never invokes caller/path callbacks under the
     run lock. No byte may be written through an unverified replacement parent. Add a hard-crash or
     response-loss retry that proves one episode, one canonical request, and byte-identical recovery.

2. **Legacy acceptance and final-writer compatibility are still neither executable nor proven / 2-of-2**
   - **Sources**: Codex standard P1; Codex adversarial Yellow 3; independently reproduced.
   - **Evidence**: the new duplicate/unknown `active_workstreams` rejection runs before checking for
     a legacy checkpoint. Existing 1.8.2 runs could legally acquire those array shapes through the
     current whitelisted state patch, so `readVerifiedState` now fails before the first checkpoint can
     normalize or migrate them. Separately, the installed-tree validator runs Task 7F proof tests
     while the Task 7B transitional direct fallback still exists, installs only `newWorkstream`
     rather than final status/terminal wrappers, and never executes legacy checker outcome, review
     completion, terminal workstream, and finish after Task 11B removes the fallback.
   - **Required response**: explicitly migrate accepted legacy active arrays in the checkpoint
     transaction without allowing the invalid shape on initialized/checkpointed runs. Install the
     final workspace writers and final no-fallback gateway in the disposable tree, then drive an old
     maker through record, checker outcome, review completion, terminal workstream, and finish.

### 🟡 Yellow — Concerns requiring response

1. **Function-valued crash seams still make mandatory recovery matrices fail / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced in the later Task 7C/7G worker diffs.
   - **Evidence**: the central gateway rejects function `crashProbe` with
     `LOCK_HELD_CALLBACK_FORBIDDEN`, but accounting and publisher crash workers still pass function
     callbacks. Those tests exit 1 before the selected journal stage rather than the asserted exit 91.
   - **Required response**: remove every function-valued crash option from public APIs, callers, and
     workers. All real crash matrices must use only the private allowlisted
     `DEEP_LOOP_TEST_CRASH_AT` scalar under `NODE_ENV=test`, and the validator must execute them.

2. **Workstream creation is not idempotent after a fully cleaned commit / solo 1-of-2**
   - **Source**: Codex standard P2; independently reproduced from the complete afterimage.
   - **Evidence**: `newWorkstream` restores `id` only through marker-backed `onRecovered`. If the
     journal has been cleaned and the response is lost, the same input allocates the next sequential
     ID and appends a duplicate workstream.
   - **Required response**: require one caller-stable bounded creation request ID, persist its digest
     in the workstream/event, reject same-ID changed payload, and return the exact prior workstream on
     a marker-free same-ID retry. Wire CLI/skills/tests to allocate once and reuse only after loss.
