## Findings

### 🔴 Red — Critical / blocking

1. **pre-marker crash recovery / corroborated — the matrix requires a marker before the publisher creates one**
   - **Sources**: Codex standard P1 and Codex adversarial Red; independently traced against the final publisher order.
   - **Location**: plan `10666-10694`, `19647-19673`, and `19740-19758`.
   - **Evidence**: `state-stage-after-rename` and `event-stage-after-rename` crash before
     `.anchored-pending.json` is durably published. The shared assertion nevertheless requires the
     marker for every crash point and then demands that same-caller/different-intent retries preserve
     every journal byte. Markerless stages carry neither caller nor intent authority and may be
     removed as orphans by the exact retry.
   - **Required response**: split pre-marker orphan-stage recovery from marker-backed recovery.
     Require intent/fence byte invariance only from `pending-after-rename` onward; prove that each
     pre-marker case has no marker, preserves canonical bytes until retry, and is cleaned and
     committed exactly once by the exact public operation.

### 🟡 Yellow — Concerns requiring response

1. **Task 11C→12B code composition / solo 1-of-2 — Task 12B redeclares or shadows descriptor locals**
   - **Source**: Codex standard P1; independently verified against sequential task application.
   - **Location**: plan `23603-23611` and `24356-24383`.
   - **Evidence**: Task 11C leaves `let action` and `let descriptorDigest` in `prepareAppTask`, while
     Task 12B inserts `const { action, descriptorDigest }`. At the same scope this is a syntax error;
     under another block it shadows the values passed into the transaction. The unconditional block
     also dereferences a null route on the manual-preserve path.
   - **Required response**: exact-replace Task 11C's provisional descriptor branch. Assign the existing
     locals only inside `route !== null`, preserve the null-route path, and make the validator reject
     the redeclaration form.

2. **await crash-worker liveness / solo 1-of-2 — its deterministic poll clock never advances**
   - **Source**: Codex adversarial Red; independently verified against the literal await loop.
   - **Location**: plan `17888-17925`, `19647-19656`, and `19911-19914`.
   - **Evidence**: the worker provides a constant `pollNowFn` and a no-op `sleepFn`, while the
     implementation derives its timeout from durable `child_ready_timeout_sec`, not the unused
     `input.timeoutMs`. The poll loop can therefore never reach the timeout append/crash probe, and
     the parent `spawnSync` has no timeout.
   - **Required response**: advance a mutable poll clock through `sleepFn`, provide an authoritative
     post-deadline mutation clock, and give every crash-worker spawn a bounded timeout with an explicit
     timeout failure assertion.

3. **Task 12B test migration / solo 1-of-2 — old expectations remain and GREEN skips the new runtime case**
   - **Source**: Codex adversarial Yellow plus main-agent test-selection verification.
   - **Location**: plan `15906-15920`, `24217-24228`, `24252-24256`, and `24385-24389`.
   - **Evidence**: Task 8B still requires an absent `descriptorBuilder` to fail and expects
     `APP_DESCRIPTOR_INVALID`, while Task 12B makes absent injection select the production builder and
     renames invalid-action failure to `APP_PREPARED_ACTION_INVALID`. In addition, Step 2 selects
     `snapshot wrapper composes` but Step 4 selects nonexistent `production prepare`, so the named
     production composition test is skipped in GREEN.
   - **Required response**: give Task 12B an exact test diff that removes the obsolete undefined-builder
     row, updates the invalid-action expectation, and uses the same `snapshot wrapper composes` pattern
     in both RED and GREEN commands. Require these tokens in the plan validator.

4. **acquire recovery intent / solo 1-of-2 — the marker binds raw observation bytes instead of normalized facts**
   - **Source**: Codex adversarial Yellow; independently verified against the design's recovery contract.
   - **Location**: design `637-645`; plan `721-756`, `11022-11047`, and `18253-18373`.
   - **Evidence**: `withAppMutation` hashes `input.observation` before `acquireAppTask` normalizes
     capability order and native path identity. Equivalent observations can therefore produce
     different marker intents and fail exact recovery even though they normalize to the same durable
     host facts.
   - **Required response**: exact-parse and normalize the observation before entering the mutation
     gateway, pass its canonical digest into the acquire intent, reuse the same normalized value in
     validation/event projection, and test marker recovery for equivalent normalized inputs plus a
     genuinely different observation.
