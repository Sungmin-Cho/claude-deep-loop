## Findings

### 🔴 Red — Blockers

1. **Episode-directory replacement can redirect request publication outside `.deep-loop` / 2-of-2**
   - **Sources**: Codex standard P1; Codex adversarial Red.
   - **Evidence**: `ensureEpisodeDirectory` validates a pathname and returns it, but
     `createFileDurablyIfAbsent` later creates `${requestPath}.create-*` and links the destination by
     pathname. Replacing the checked episode directory with a symlink between those operations
     redirects the temporary write itself outside the run. Existing tests install a symlink before
     validation or cover only an `EEXIST` winner; neither exercises the check-to-create race.
   - **Required response**: publish through an atomically bound non-symlink directory identity, or
     use a staging-directory/no-replace protocol whose first write cannot traverse the replaceable
     episode path. Fail closed where the required primitive is unavailable. Inject the swap exactly
     between validation and first creation and prove the external target remains byte-for-byte
     untouched.

2. **The claimed legacy lineage checkpoint is absent from the literal verifier / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced from the Task 7A verifier.
   - **Evidence**: the Task 7A test requires duplicate and corrupted
     `lease-lineage-baselined` events to fail, but `verifyAppEventCorrelation` walks acquisition
     edges toward generation 1 without reading that event. On an initialization-absent run it drops
     incomplete lineage without adding an error. The separate episode verifier then consumes the
     same checkpoint's episode count/digest, creating split proof semantics.
   - **Required response**: validate exactly one bounded checkpoint with exact keys, lease identity,
     timestamp/order, pre-state digest, and complete legacy episode projection; trace only exact
     post-checkpoint generation edges. Execute the real Task 7A checkpoint cases in the sequential
     installed-tree validator rather than syntax-checking the helper alone.

### 🟡 Yellow — Concerns requiring response

1. **Same-ID review replay still depends on mutable checker and discovery state / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced.
   - **Evidence**: `replayReviewDispatch` compares immutable `dispatch_response.reason` with the
     current `checker.block_reason`, which the real claim-to-block transition mutates. The complete
     dispatch digest also includes current plugin detection before durable same-ID recovery. A
     response-loss retry can therefore throw after block transition or detector drift. The round-4
     regression changed contract/insights only and edited status directly, so it did not close these
     paths.
   - **Required response**: authenticate same-ID replay from caller-stable identity before plugin,
     contract, insights, or mutable checker state. Reconstruct only from immutable dispatch fields.
     Add real claim→block and changed-detection same-ID replay tests.

2. **Mandatory review request IDs are not migrated through every caller / solo 1-of-2**
   - **Source**: Codex standard P1; independently reproduced against the baseline call inventory.
   - **Evidence**: the plan makes `dispatchReview.requestId` and CLI `--request-id` mandatory, while
     the canonical Route A–C commands in `skills/deep-loop-workflow/references/adapters.md` and many
     direct calls in `tests/review*.test.mjs`, `tests/finish.test.mjs`, `tests/fencing.test.mjs`, and
     adjacent suites remain unchanged. The planned full suite would fail with usage or
     `REVIEW_DISPATCH_REQUEST_ID_REQUIRED` errors.
   - **Required response**: migrate every production, skill/docs, and test caller through one stable
     allocate-once/reuse-on-response-loss convention. Add a validator inventory that rejects any
     unapproved direct call or canonical CLI example lacking the ID.

3. **App emit uses lossy numeric native file identities on Windows / solo 1-of-2**
   - **Source**: Codex standard P2; independently reproduced.
   - **Evidence**: the final `appNativePathDeps` factory uses plain `statSync`, unlike the other
     native identity factories. Device/inode values above `Number.MAX_SAFE_INTEGER` can round to the
     same Number while `sameNativeDirectory` accepts equality without canonical-path equality on
     Windows.
   - **Required response**: use `statSync(value, { bigint: true })` and retain the adjacent
     above-safe-integer collision regression on this final production factory.

4. **The final App status shape leaves an earlier exact-key assertion stale / solo 1-of-2**
   - **Source**: Codex adversarial Yellow; independently reproduced.
   - **Evidence**: the fixed interface and final Task 9B implementation return `resume_policy`, but
     the retained Task 6D `Object.keys(status).sort()` assertion omits it. The full test suite would
     fail after the later task adds the field.
   - **Required response**: migrate the earlier assertion in the task that introduces the final
     shape and add exact complete projections for `resume_policy='app'`, `'human'`, and `null`.
