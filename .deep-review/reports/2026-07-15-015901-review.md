## Findings

### 🔴 Red — Critical / blocking

1. **error-handling / corroborated — public mutation retries read a pending marker before entering recovery**
   - **Source**: Codex standard P1 and Codex adversarial HIGH.
   - **Location**: plan `10017-10020`, `10051-10073`, `13756-13764`, `14881-14918`,
     `15508-15526`, `15570-15596`, `16132-16155`, `16706-16730`, `17756-17777`,
     `18148-18164`; design `603-614`.
   - **Evidence**: `readVerifiedState` intentionally rejects `.anchored-pending.json`, while
     matching recovery starts only inside `withVerifiedMutationLock`. `emitHandoff`, prepare,
     confirm/fail, sweep/await, acquire, recover, and finish all perform a public verified read before
     their eventual `appendAnchored` gateway. A crash after marker publication therefore makes exact
     retry return `ANCHORED_TRANSACTION_PENDING` rather than recover and re-enter idempotency.
   - **Required response**: make the recovery gateway own each complete mutating operation, including
     its first canonical read and idempotency decision; expose only an opaque lock-owned read/commit
     context, reject same-caller/different-intent marker recovery, and exercise the real public APIs at
     every crash point.

### 🟡 Yellow — Concern

1. **test-order / verified — Task 7B requires operations that do not yet exist**
   - **Location**: plan `9893-9916`, `10461-10481`, `14087-14097`, `15020-15028`,
     `16394-16402`.
   - **Evidence**: Task 7B's worker enum and GREEN/full-suite claim include prepare, confirm, and
     acquire, but those exports are first introduced in Tasks 8B, 9A, and 10A. Sequential TDD cannot
     execute Task 7B as written.
   - **Required response**: retain only already-existing generic gateway operations in Task 7B and
     place the full real-public-operation crash matrix after the final App mutation APIs exist.

2. **test-fixture / verified — Task 3B's own valid fixtures violate its exact projection schema**
   - **Location**: plan `1705-1814`, `2540-2644`.
   - **Evidence**: `appSchemaLoop` and all manual profiles store `request_projection: {}` and expect
     `validate(...).ok === true`; the same task requires the projection to have exactly 12 top-level
     keys and fixed nested shapes. The normalizer that could create that shape is not added until
     Task 4A.
   - **Required response**: add a complete local exact-projection fixture helper in Task 3B and use it
     for every initialized schema fixture.

3. **plan-executability / verified — Task 7G has no literal test or implementation code**
   - **Location**: plan `374`, `13101-13160`, `25732-25735`.
   - **Evidence**: the card's Step 1 and Step 3 contain only prose and `text` checklists, although the
     plan contract requires directly applicable complete tests and implementation. The embedded
     validator counts `text` as a complete code fence, so it falsely accepts this card.
   - **Required response**: add executable JavaScript tests and implementation diffs for the seven
     publisher migrations, and make the validator require executable-language or diff fences for
     code-producing Step 1/3 blocks.

4. **privacy / verified — message-unconfirmed proof allows extra raw receipt fields**
   - **Location**: plan `8877-8885`, `8929-8947`; design `392`.
   - **Evidence**: confirmed events enforce an exact key set, but the `message-unconfirmed` verifier
     checks only its digest. A hash-valid rewritten failure event can include raw
     `unconfirmed_thread_id` alongside the correct digest and still pass.
   - **Required response**: enforce an exact event-data key set for every failure subtype and add a
     hash-valid negative fixture with an extra raw-ID field.
