## Findings

### 🔴 Red — Independently corroborated blocker

1. **Episode recovery overwrites the execution plane's editable `request.md` / 2-of-2**
   - **Sources**: Codex standard P2; Codex adversarial Yellow.
   - **Evidence**: `onExisting` and marker recovery reconstruct the existing episode path, but
     `createEpisode` ignores the returned replay result and falls through to the unchanged
     unconditional `atomicWrite(requestPath, requestSkeleton(...))` tail.
   - **Counterexample**: after the execution plane fills the request task, a delayed exact dispatch
     retry replaces it with the original skeleton even though no new episode is created.
   - **Required response**: distinguish created/recovered materialization, preserve every existing
     editable request byte, materialize only a genuinely missing file, and test both preservation and
     missing-file crash recovery.

### 🟡 Yellow — Concerns requiring response

1. **Content-only episode identity conflates a later new episode with an old retry / solo 1-of-2**
   - **Source**: Codex standard P1; independently reproduced from `episodeRequestDigest` and
     `onExisting`.
   - **Evidence**: the digest contains the normalized episode content but no caller-supplied logical
     request identity. A later intentional `newEpisode` with identical arguments therefore reuses the
     first episode forever.
   - **Required response**: require and persist a bounded logical creation request ID, domain-separate
     its digest from the full payload digest, reject changed reuse, and migrate every caller.

2. **Terminal checker-receipt no-op regression is not preserved / solo 1-of-2**
   - **Source**: Codex standard P1; independently reproduced against
     `tests/budget.test.mjs:1199` and the planned `assertProcessReplayPolicy`.
   - **Evidence**: after handoff, child acquisition, and terminal finish, the accepted current child
     fence is deliberately different from the checker receipt's parent origin. The planned terminal
     checker verifier still requires lease owner/generation to equal that old origin, so the existing
     exact retry cannot remain a write-free no-op.
   - **Required response**: preserve the current-child authorization plus immutable parent-origin
     receipt proof as distinct identities and migrate the existing regression into the sequential
     afterimage validator.

3. **Breaker verdict replay ignores `review-outcome` counter transitions / solo 1-of-2**
   - **Source**: Codex standard P2; independently reproduced from `commitReviewOutcome` and
     `recoveredReviewVerdict`.
   - **Evidence**: replay reconstructs the counter from only `breaker-reset` and
     `breaker-review-verdict`, while `review-outcome` still increments or resets the same durable
     counter.
   - **Required response**: consolidate the mutation event path or include authenticated
     `review-outcome` transitions in the lineage and cover interleavings in both directions.

4. **A crashed nested review creation cannot reach its recovery intent / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently traced through `dispatchReview` and the mutation
     gateway.
   - **Evidence**: public retry first enters intent `dispatchReview`, but the pending marker was
     published by distinct intent `dispatch-review-episode`; the gateway rejects the mismatch before
     the body can enter the recovering phase.
   - **Required response**: make dispatch phase-aware before outer gateway entry or use one complete
     public-operation intent, then crash-test every journal point through the public API only.

5. **Task 7G's combined JavaScript afterimage is not executable or validator-applied / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced from Task 7G Step 3 and the validator.
   - **Evidence**: one unscoped JavaScript fence contains both lease and breaker bodies, while
     `exactLeaseFence` remains private to the lease portion and is referenced by the breaker portion.
     The sequential validator applies only the breaker import diff and syntax-checks the combined
     fence without installing it.
   - **Required response**: provide separate complete lease and breaker patches with local/exported
     helpers, apply both production and test afterimages sequentially, and run syntax plus focused
     tests on that disposable tree.

6. **Terminal reset precedence contradicts the proposed implementation / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced from the Task 7G terminal test and
     `resetBreaker` afterimage.
   - **Evidence**: the test expects `LEASE_FENCED: RUN_TERMINAL`, but the implementation's
     identity-only fence can return `alreadyReset` before `leaseCheck` enforces terminal policy.
   - **Required response**: retain identity-first authentication, then run the complete current
     `leaseCheck` before no-op detection and lock the exact error precedence in the applied test tree.

7. **`tripBreaker` and `resetBreaker` lack durable request identity / solo 1-of-2**
   - **Source**: Codex adversarial Red; independently reproduced from their API signatures and
     `breakerMutation`.
   - **Evidence**: only verdict recording receives a request ID. A lost reset response followed by a
     newer verdict can be retried as a fresh reset and erase the newer transition; immediate recovery
     also returns a different projection.
   - **Required response**: require stable request IDs for all three breaker APIs, persist
     domain-separated ID and full-request digests, recover the original projection before current
     no-op evaluation, and test intervening-operation retries.

8. **Episode state identity is not correlated with its immutable event / solo 1-of-2**
   - **Source**: Codex adversarial Yellow; independently reproduced from Task 7F and
     `verifyRunSnapshot`.
   - **Evidence**: state and event each store a request digest, but semantic integrity never proves
     one exact `episode-new` event corresponds to one episode object and immutable initial projection.
   - **Required response**: add cross-log episode creation correlation and corruption tests for
     state-only and event-only mismatches.

9. **Delayed review-dispatch retry still fails after checker transition / solo 1-of-2**
   - **Source**: Codex adversarial Yellow; independently reproduced from the current first-phase maker
     eligibility filter.
   - **Evidence**: the plan excludes current checker status from immutable recovery identity, but
     `dispatchReview` rejects makers whose checker has already approved or rejected before performing
     request-ledger lookup.
   - **Required response**: perform exact request-ledger replay lookup before fresh maker eligibility,
     while still enforcing current lease/terminal policy; cover pending, in-progress, approved,
     rejected, and blocked retries.
